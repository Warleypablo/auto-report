# F5 — Atualizar gestores: re-sync ClickUp sobrescrevendo + gestor_travado + botão Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (- [ ]) syntax for tracking.

**Goal:** Fazer o re-sync de gestores do ClickUp **sobrescrever** `clientes.gestor` quando o responsável mudou na fonte — exceto quando `clientes.gestor_travado = true` (proteção de edição manual). Expor a flag `gestor_travado` na API de edição de cliente e no payload de listagem; adicionar um toggle "Travar gestor" no modal de edição de cliente e surface do número de elegíveis (`a_atualizar`) no botão/resultado "Sincronizar gestores".

**Architecture:**
- O endpoint `POST /gestor/clientes/sync-gestores` (em `web/backend/api/gestor.py`, função `sync_gestores_from_clickup`, linhas 385–534) **já** sobrescreve `clientes.gestor` via `UPDATE clientes c SET gestor = pr.novo_gestor ... WHERE c.id = pr.cliente_id AND (c.gestor IS DISTINCT FROM pr.novo_gestor)` (linhas 512–517). F5 adiciona `AND c.gestor_travado = false` nesse `UPDATE` (onde `c` é a row alvo de `clientes`, então a coluna é referenciável diretamente) **e** na pré-contagem `a_atualizar` (linhas 462–464, que opera sobre o CTE `performance_resp` — precisa propagar `gestor_travado` pelos CTEs), e devolve `a_atualizar` na resposta. O caminho de dados (match por nome normalizado → `subtask_ids` → `cup_contratos` filtrando `servico ILIKE '%performance%'` → `responsavel`) permanece igual.
- A coluna `clientes.gestor_travado` (`Boolean`, `default false`, `server_default "false"`) é criada na **F0** (Contrato §1.3 e §8) — **pré-requisito desta fase**. F5 apenas a consome (não cria migration própria — Contrato §8).
- `gestor_travado` é exposto em `ClienteGestorItem`, `ClienteDetalheItem` e aceito em `ClienteEditRequest` (Pydantic), então o `PATCH /gestor/clientes/{cliente_id}` (handler `update_cliente`, linhas 884–912, loop genérico `setattr` com `model_dump(exclude_unset=True)`) passa a setá-lo, e o `GET /gestor/clientes` o devolve.
- Frontend: a UI real de edição de cliente é um **modal de edição** dentro de `web/frontend/app/gestor/page.tsx` (tipo `EditForm` na linha 1105, `emptyForm` na 1118, `handleSalvar`/`payload` na 1307, JSX do modal na 1656). F5 adiciona `gestor_travado` ao tipo `ClienteGestor`/`ClienteEditData` (`lib/api-gestor.ts`), ao `EditForm`/`emptyForm`/`payload`, e renderiza um toggle (checkbox) dentro do modal que envia o campo no `PATCH`. A página de vínculos ClickUp (`app/gestor/admin/clickup-vinculos/page.tsx`) já tem o botão "Sincronizar" (gestores) — F5 enriquece o estado/resultado com `a_atualizar`.

> **Nota de correção do rascunho:** NÃO existe `web/frontend/app/gestor/admin/clientes/page.tsx`. A edição de cliente vive em `web/frontend/app/gestor/page.tsx`. O alvo de UI desta fase é o **modal de edição** desse arquivo (não uma lista com botão por linha).

**Tech Stack:** Backend Python 3 / FastAPI / SQLAlchemy 2.x / Pydantic v2 / pytest + `TestClient` (DB `vitrine_test` via `TEST_DB_URL`, `Base.metadata.create_all`). Frontend Next.js / TypeScript / Playwright.
